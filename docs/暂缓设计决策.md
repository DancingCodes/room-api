# Deferred Design Decisions

本文记录本轮设计评审中已经讨论过、但明确暂不修改的点。后续除非产品或部署条件变化，否则不要把这些点当作待修 bug 反复处理。

## 明确暂不修改

- HTTP 错误响应仍统一返回 HTTP 200，业务错误继续通过响应体 `code/message/data` 表达。
- WebSocket 断开仍等价于离开房间，不拆分“在线连接状态”和“房间成员关系”。
- WebSocket 客户端发送队列满时，仍采用关闭连接并移除客户端的保护策略。
- 暂不引入结构化业务错误类型；handler 继续按当前方式返回业务错误。
- 端口和运行参数暂不配置化，服务端口继续按当前实现处理。
- 源码中文乱码显示问题不作为代码问题处理；这是读取/终端编码问题。
- 验证码暂不改为哈希存储。
- 验证码暂不调整为先入库再发送，继续保持当前发送和写入顺序。
- 唯一键冲突暂不做更细的业务错误映射，继续依赖数据库唯一索引兜底。
- 暂不增加数据库外键约束，关联数据一致性继续由业务代码维护。
- 头像更新后暂不清理旧 COS 对象。
- JWT 暂不增加主动失效、版本号、黑名单或刷新机制。
- 注册前公开头像上传入口暂不改。
- 注册仍要求头像 URL，不改为默认头像或注册后补头像。
- 用户同一时间只能在一个房间的约束暂不改。
- 消息发送暂不增加客户端消息 ID 或幂等机制。
- 登录、注册、上传、发消息、进房间等接口暂不增加应用级限流。
- 服务启动和退出暂不改为优雅关闭流程。

## 重新评估触发条件

- 准备公网大规模开放时，应重新评估限流、公开上传、验证码存储和登录安全策略。
- 准备支持多端、房间旁听、断线重连保留席位时，应重新评估“WebSocket 断开即离房”和“用户只能在一个房间”的约束。
- 准备做消息本地状态、失败重试、离线重放时，应重新评估消息幂等机制。
- 存储成本明显增长时，应重新评估旧头像对象清理策略。
